# Security, Secrets & Compliance

Security is not an afterthought; it is built into our daily development and deployment workflows.

## 1. Secrets Management

We enforce a strict policy regarding API keys, database passwords, and private keys: **NEVER commit secrets to Git.**

### Guidelines
* **Use Environment Variables:** All configuration settings and credentials must be loaded via environment variables (using `.env` files locally and secure environment injection on our servers).
* **Gitignore:** Ensure that `.env`, private keys, and local config files are always added to `.gitignore`.
* **GitHub Secrets:** All CI/CD credentials must be stored securely in GitHub Secrets, never hardcoded in YAML files.

## 2. Server & File Permissions

On our OpenLiteSpeed VM, we enforce strict Unix file permissions to prevent unauthorized access or script execution:

* **Directories:** Must be set to `755` (owner can read/write/execute, others can read/execute).
* **Files:** Must be set to `644` (owner can read/write, others can read).
* **wp-config.php:** Must be set to `600` or `400` to prevent other system users from reading database credentials.

```bash
# Secure wp-config.php permissions
chmod 600 /var/www/myrehat/wp-config.php
```

## 3. WordPress Security Hardening

WordPress is a frequent target for automated scanners and exploits. We harden our installation with these standard rules:

1. **Disable XML-RPC:** XML-RPC is a legacy feature frequently exploited for brute-force and DDoS attacks. We disable it in our custom redirect plugin or Nginx config.
2. **Hide WordPress Version:** Prevent scanners from identifying outdated core versions by removing generator tags.
3. **Restrict File Editing:** Disable the built-in WordPress theme and plugin editor to prevent administrative users from executing malicious PHP code if their account is compromised:

```php
// Add to wp-config.php
define('DISALLOW_FILE_EDIT', true);
```
